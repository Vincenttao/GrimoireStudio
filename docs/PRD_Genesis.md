**版本号:** v4.1 (Story Gods Paradigm)
**产品代号:** Project Genesis
**文档状态:** Draft / Vision Review

## 1. 背景和原理 (Background & Principles)

### 1.1 痛点与破局
当前市面上的 AI 写作或互动工具（如 Character.AI 或各类 Copilot）大都深陷于"黑盒打字机"或单调的"角色陪聊"范式中，带来了三大终极痛点：
1. **角色崩坏 (OOC)**：随着剧情深入，AI 容易产生幻觉，遗忘初始设定与核心动机。
2. **节奏拖沓 (缺乏张力)**：AI 习惯顺延上文生成，缺乏主动把握戏剧冲突的能力，极易产生"流水账"。
3. **质感平庸 (缺乏视角与文风)**：现有生成工具无法区分"上帝视角"和"限制视角"，心理描写过于直白，毫无文学留白的美感。

**核心愿景：**
抛弃"AI 代笔 (Copilot)"，拥抱"自主叙事引擎 (Autonomous Narrative Engine)"。用户不再辛苦敲字，而是降临为"上帝 / 导演"，由后台 AI 角色们自动演戏，从而涌现出极具真实感的高能剧情。

### 1.2 运作原理：管线解耦与七大 Agent
Genesis Engine 将文学创作解耦为**"发生（剧情逻辑）"**与**"表达（文学渲染）"**两个完全独立的阶段。系统由以下七个核心 Agent 协作驱动：

| Agent | 角色 | 核心职责 | 输入 | 输出 |
|---|---|---|---|---|
| 🧚‍♀️ **The Muse** | 责编/助理 | 面向用户的唯一总代理 (UI Proxy)。翻译口语为底层操作，通过 Markdown 对话确认，拒绝弹窗。 | 用户自然语言 | 结构化操作指令 |
| 🎭 **Character** | 演员 | 每个角色一个独立 AI，基于隐藏动机在沙盒中博弈。 | 场景设定 + Grimoire 约束 | 角色意图与动作 |
| 🌍 **World** | 裁判 | 查阅 The Grimoire 判定动作是否合法，防止角色"开挂"。 | 角色动作 + Grimoire | 合法/非法裁决 |
| 👑 **Showrunner** | 导演 | 监控推演张力，僵局时主动下发突发事件 (Plot Twist) 打破平衡。 | 推演进程 | 导演指令/终止信号 |
| ✂️ **Director** | 剪辑师 | 将冗长的推演记录"脱水"为严谨结构化的剧本中间件 (**Story IR**)。 | 原始推演日志 | Story IR Block |
| 🎥 **Camera** | 渲染器 | 唯一负责文学修饰。根据 POV 和文风参数，将 Story IR 渲染为最终文字。 | Story IR + 渲染参数 | 文学正文 |
| 📜 **Scribe** | 史官 | 严格基于 Story IR（非渲染文本）提取事实，回写至 The Grimoire，防止幻觉污染。 | Story IR Block | Grimoire 更新 |

## 2. 界面布局与导航 (UI Layout & Navigation)

### 2.1 整体布局：三栏 + 导航条

![Genesis Engine 3-Pane UI Layout](/docs/genesis_engine_final_ui_1773144668812.png)

系统采用**极简的三栏布局**，UI 交互的最高原则为 **"永不弹窗遮挡、专注唯一稿纸"**。

| 区域 | 名称 | 占比 | 核心功能 |
|---|---|---|---|
| 最左侧极细导航条 | Navigation Strip | ~3% | 切换四大全局视图 |
| 左栏 | The Compass | ~20% | 大纲树 (Storyboard) + 设定树 (Grimoire) |
| 中栏 | The Manuscript | ~52% | 正文编辑器 (上) + 推演监视器 (下) |
| 右栏 | The Muse | ~25% | AI 助理聊天面板（从上到下完整占满） |

### 2.2 四大全局视图 (Navigation Strip)
最左侧的导航条包含四个图标，点击后**切换中栏和左栏的内容**，右栏 The Muse 始终常驻。

| 图标 | 名称 | 功能描述 | 用户可执行的操作 |
|---|---|---|---|
| 🧭 | **Compass (罗盘)** | **默认主工作台**。左栏显示大纲树和设定树，中栏显示正文编辑器和推演监视器。 | 浏览/展开大纲和设定，编辑正文，启动推演 |
| 🕸️ | **Network (关系网)** | **全局实体关系可视化**。中栏替换为力导向图 (Force Graph)，展示角色、势力、地点的动态关系。 | 查看关系、点击节点高亮关联实体、按章节时间线筛选关系快照 |
| 📚 | **Archive (档案馆)** | **全局检索与历史**。中栏替换为搜索界面，支持按角色名、关键剧情事件、章节范围进行全文检索。 | 搜索历史 IR 记录、查看废弃的平行宇宙分支、浏览系统操作日志 |
| ⚙️ | **Settings (设置)** | **项目配置**。中栏替换为设置面板。 | 管理 API 密钥、配置模型路由策略（High/Low）、查看 Token 消费统计看板 |

### 2.3 左栏交互规则

**The Storyboard (大纲树)：**
*   点击某个 `Chapter` 节点 → 中栏 (The Manuscript) **瞬间无缝切换**为该章节文本。
*   点击某个 `Branch` 节点 → 中栏原地切换为该平行宇宙分支的文本。
*   所有切换均为原地替换，用户始终面对"唯一一张稿纸"。

**The Grimoire (设定树)：**
*   单击某个实体节点（角色/地点/势力）→ 节点在左栏**原地向下展开 (Accordion Inline Peek)**，滑出微缩属性卡片（头像、核心属性、当前状态）。中栏纹丝不动。
*   所有设定的修改必须通过 The Muse 的对话确认完成，设定树本身为**只读展示**。

## 3. 创作工作流与用户体验 (The Genesis User Journey)

### 3.1 场景一：零基础开局 (The Cold Start)
对于有着顶级脑洞但厌恶写大纲的作者来说，面对空白项目是最痛苦的。在 Genesis Engine 中，由于 **The Muse Agent** 的存在，这个过程被降维成了一场轻松的"贴身责编催稿局"。

1.  **灵感碰撞 (The Brainstorm)**：用户在 **The Muse** 对话框中直接输入口水话："我想写个赛博朋克修仙。主角是个搞笑道士，修仙靠赛博义体和物理超度。" **The Muse** 会高速解析并引导用户确认核心冲突与主角动机。
2.  **世界观暗中固化 (The Grimoire Injection)**：**The Muse** 在对话流中用 Markdown 格式展示属性修改摘要并询问确认。用户回复"确认"后，属性即刻入库至 **The Grimoire**。
3.  **大纲枝干生成 (The Storyboard Generation)**：**The Muse** 协助构思第一章冲突。确认后，左侧 The Storyboard 树上瞬间挂载新章节，并生成一个 **The Spark**（火花），等待启动推演。

### 3.2 场景二：日常连载流 (The Serialized Workflow)
由于采用了**边演边建 (Lazy Instantiation)** 策略，不需要先建完世界观，就可以开始常态化的连载循环。

#### 阶段一：The Grimoire (世界观状态继承)
作者登入系统，左侧栏的 Grimoire 已自动更新为最新的世界状态。所有属性均由 **Scribe Agent** 在上一次提交后自动从 Story IR 中提炼回写，保证角色绝不 OOC。

#### 阶段二：The Spark (创意火花)
**定义：** The Spark 是一段**自然语言冲突指令**，它是启动一次推演的钥匙。

| 属性 | 说明 |
|---|---|
| **输入位置** | 可在中栏 Storyboard 新节点中直接输入，也可通过 The Muse 对话代为生成 |
| **内容** | 一段口语描述（例："乌家截断了蚕丝，苏檀儿急疯了向宁毅求助，我要宁毅用极度散漫的态度随口给出一条毒计"） |
| **系统行为** | Muse 解析指令，自动从 Grimoire 中拉取相关角色和场景约束，组装后下发至推演沙盒 |
| **触发** | 用户点击 **[Action ▶]** 按钮，推演启动 |

#### 阶段三：The Monitor (推演态：异步回合制)
后台的 Agent 们开始基于当前世界状态的沙盒推演。推演结果以**异步回合制**展示在中栏底部的 Monitor 面板中。

**回合 (Turn) 定义：**
*   一个 Turn = 一个角色输出一次完整的意图与动作 + World Agent 对该动作的合法性裁决。
*   每个 Turn 完成后，系统暂停 2-3 秒（可在 Settings 中配置），供用户检视。
*   如果用户不干预，倒计时结束后自动进入下一个角色的 Turn。

**场景结束条件 (Beat Completion)：**
*   **Showrunner 自主判定**：当场景张力达到预期高潮（如核心冲突已揭示、反转已完成），Showrunner 发出终止信号，推演收束。
*   **用户手动终止**：用户随时可以点击 **[Cut ✋]** 按钮，强行终止当前推演段落。

**🕊️ 上帝之手 (God's Hand)：**
用户随时可以点击 Monitor 中某条推演记录：
1.  推演时间线**瞬间冻结**，所有 Agent 暂停。
2.  用户可以**覆写**该角色的意图（如编辑文本："让宁毅态度再狂一点"）。
3.  用户点击 **[Release ▶]**，时间线恢复。后续所有 Agent 的反应将基于覆写后的新设定自动重新推算。

**异常流：**
*   **World Agent 裁定非法**：Monitor 中该条动作会被标记为 ❌，附带裁判理由（如"角色当前没有该武器"）。系统自动要求该 Character Agent 重新生成合法动作。用户也可以选择点击 **[Override (上帝特赦)]** 强行通过。

#### 阶段四：IR 脱水与锁定 (Director 介入)
推演结束后，**Director Agent** 自动将冗长的推演记录"脱水"为严谨的结构化剧本中间件 (**Story IR Block**)。此时剧情骨架已锁死，不可篡改。此过程对用户透明。

#### 阶段五：全局渲染与局部精修 (The Render & Revision)

**全局渲染台 (Render Mixer)**：位于中栏 The Manuscript 的**右上角常驻入口**。

| 参数 | 类型 | 说明 |
|---|---|---|
| **视点 (POV)** | 下拉选择 | 可选值：某特定角色的限制视角、全知视角（上帝视角）、第一人称 |
| **文风 (Style)** | 下拉 + 自定义 | 内置预设模版（如"商战/权谋 肃杀风""仙侠/飘逸 古典风"）；支持用户粘贴一段参考文段作为自定义文风锚点 |
| **潜台词 (Subtext)** | 滑块 (0-100%) | 控制留白与心理描写的密度。0% = 纯白描，100% = 极致意识流 |

*   **全局渲染:** 设定以上参数后，点击 **[Render All]**，Camera Agent 将整章 IR Block 一次性渲染为风格统一的正文。
*   **🔍 局部批改 (Block Revision)**：选中某段不满意的文本 → 弹出 **[Revision (批改)]** 悬浮菜单 → 输入修改指令（如"增加潜台词描写"）→ 仅该段文字在原位无损重写，不影响前后文风。

**异常流：**
*   **渲染质量不佳**：用户对整段渲染不满意时，可点击 **[Retry]** 重新渲染。由于底层 IR 不变，可以无限次重试且无副作用。
*   **致命质量问题**：如果多次 Retry 仍不满意，用户可在 Muse 中沟通调整渲染参数策略，或切换底层渲染模型。

#### 阶段六：确认与记忆回写 (The Commit)
1.  用户阅读成文满意后，点击 **[Commit (锁定)]**。
2.  **Scribe Agent** 严格读取阶段四的 Story IR（注意：绝不读取阶段五渲染后的文学散文，以避免幻觉污染），提取最新事实，回写至 The Grimoire。
3.  创作闭环完成。用户继续抛出下一个 Spark，推进今日的第二个章节段落。

### 3.3 The Muse 可执行操作清单 (Command Matrix)
**The Muse** 不只是聊天机器人，它是系统暴露给用户的 **终极交互代理 (UI Proxy)**。

| 操作类别 | 示例用户指令 | Muse 行为 | 确认级别 |
|---|---|---|---|
| **创建实体** | "加一个新角色叫张三，是个剑客" | 生成属性摘要，Markdown 展示 | 用户回复"确认"后写入 |
| **修改实体** | "给宁毅加个设定，他其实有点怕老婆" | 展示修改 Diff 摘要 | 用户回复"确认"后写入 |
| **删除实体** | "把那个路人甲删掉" | 展示即将删除的实体信息 | **二次确认**（"这将永久删除，确定吗？"） |
| **生成 Spark** | "帮我想一个今天的冲突点" | 基于 Grimoire 当前局势推演三个选项 | 用户选定后自动填入 Spark |
| **微操推演** | "让宁毅态度再狂一点" | 解析指令，冻结并覆写当前推演角色意图 | 直接执行 |
| **调整渲染** | "潜台词再多一点" | 调整 Render Mixer 参数 | 直接执行 |
| **查询记忆** | "宁毅和苏檀儿现在是什么关系？" | 查阅 Grimoire 返回当前状态 | 无需确认（只读） |
| **分支管理** | "在第490章切一个新分支" | 展示即将创建的分支信息 | 用户确认后执行 |
| **合并章节** | "把第三章和第四章合并" | 展示合并预览 | 用户确认后执行 |
| **回档** | "回到第200章的状态" | 展示回档影响范围 | **二次确认** |

### 3.4 全本协作的终极奥义 (The Long Game)
1. **世界观自动生长**: The Grimoire 随着每一次推演自动更新，它是真正"边演边建"的，作者无需手动维护人物关系表格。
2. **多分支树状宇宙 (Branching Tree UI)**: 作者可以随时在几百章之后**切出一个新分支（平行宇宙）**，尝试新的剧情走向，且一键回档，所有底层关系网一并重置。不同分支互不干扰。
3. **出书级一键渲染**: 连载结束后，一键修改全局渲染参数（如切为"第三人称全知/剧本对白风"），即可得到不同媒介（剧本、实体书）的全本内容。无需重构一丝一毫的故事逻辑。

## 4. 版本路线图 (Release Roadmap)

### 4.1 V1.0：The Core Engine (核心引擎)
*   **目标:** 跑通"推演 → 脱水 → 渲染"的核心数据链路，验证管线解耦的可行性。
*   **交付物:**
    *   Story IR 的结构化提取与校验系统。
    *   Block 排序引擎（支持追加与任意插入）。
    *   Scribe 史官记忆系统（Story IR → The Grimoire 的自动提取与覆写）。
    *   Camera Agent 基础渲染管线。
*   **用户感知:** 用户在一个相对原始的界面中输入设定 (Spark)，后台通过日志展示推演过程，编辑器中直接呈现渲染后的文字。没有华丽的动画，**但逻辑已经能支撑写完一本连贯的小说。**
*   **验收标准:** 能在系统内完成一个"三角色、五章节"的连贯短篇，Scribe 提取的世界状态在每章结束后 100% 准确。

### 4.2 V2.0：The Co-Pilot (造梦控制台)
*   **目标:** 在坚实的数据底座上重构 UI 交互，让用户真正感受到"做上帝"的爽感。
*   **交付物:**
    *   🧚‍♀️ **The Muse Agent** 上线，支持 Command Matrix 中全部操作。
    *   **The Monitor 异步推演监视器** 上线，包含回合制展示、上帝之手（冻结/覆写）、上帝特赦。
    *   **The Render Mixer** 上架至正文区右上角（含 POV / Style / Subtext 三参数）。
    *   三栏极简 UI 布局落地。
*   **用户感知:** 系统从"命令行工具"变成了一个类似《博德之门3》掷骰子的"沉浸式文字沙盒游戏"。普通网文新手也能通过跟 Muse 聊天轻松驾驭引擎。
*   **验收标准:** 一个对系统零基础的网文作者，仅通过跟 Muse 对话，能在 30 分钟内完成从"冷启动"到"产出第一章 2000 字正文"的完整流程。

### 4.3 V3.0：The Studio (专业工作站)
*   **目标:** 完备长篇/剧本的专业工作站，处理百万字级别的超大项目。
*   **交付物:**
    *   **Branching Tree UI（多分支漫游图）**：支持从任意章节切出平行宇宙分支。
    *   🕸️ **Network 关系网视图** 上线。
    *   📚 **Archive 档案馆** 全文检索上线。
    *   一键全本导出，支持小说 / 剧本 / 大纲等多格式分发。
*   **用户感知:** 面对 300 万字的小说也不会迷路。作者可以自由穿梭在平行分支之间，一键回档，并最终打包出版。
*   **验收标准:** 一个包含 50+ 章节、3 个平行分支的项目可以流畅运行并一键导出。

## 5. 非功能性需求 (Non-Functional Requirements)

1. **多模型切分降本 (High/Low Routing):** 推演/记忆层可使用高速低成本模型（如 DeepSeek 等），渲染/剪辑层路由至顶级模型保证质感。在 Settings 中提供可视化 Token 消费看板。
2. **零硬编码 (No Hardcoded Prompts):** 所有 Agent 的性格设定与工作指令均为独立模板文件集中维护，可随时热更换而无需重新部署。
3. **不可变性与回档:** 每一次推演和渲染都产生新的修订版，天然具备版本控制能力。绝不覆盖历史数据。
4. **性能:** The Monitor 推演日志应以亚秒级延迟到达前端。渲染 2000 字正文不超过 15 秒。
5. **安全:** Human-in-the-loop 是底线。任何对 The Grimoire 的写入操作必须经过用户在 Muse 对话中的显式确认。
