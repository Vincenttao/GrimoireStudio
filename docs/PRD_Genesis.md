## 0. 北极星用户与产品定位（NEW in v5.0）

### 0.1 北极星用户

**中文网文连载作者。** 详见 `USER_PERSONA.md`。没有次要人群。

所有功能必须回答："**这能帮一个网文作者每天稳定日更 3000+ 字吗？**" 回答不出来的不做。

### 0.2 产品核心承诺

> **Genesis Engine 不是 AI 写作工具，是网文作者的"日更作坊"。**
>
> - 你抛冲突，AI 演戏，Camera 渲染，一小时一章。
> - 角色永不 OOC（Grimoire + VoiceSignature + 快照链）。
> - 字数精准命中平台要求（Camera target_char_count 约束）。
> - 章末钩子自动守卫（Ending Hook Guard）。
> - 卡文一键救急（`[卡文救急]` + 3 个 Spark 候选）。
> - 单章成本 ¥1 以内（默认 Qwen + DeepSeek 组合）。
> - 作品就是一个 `.sqlite` 文件，100% 本地所有。

### 0.3 反定位（不做什么）

| 明确不做 | 原因 |
|---|---|
| 英文写作 | 所有 Prompt 和预设为中文特化 |
| 剧本 / 非虚构 | 架构不匹配 |
| 多人协作 | 单用户单体是架构承诺 |
| 云端托管作品 | 本地所有权是用户最大卖点 |
| 严肃文学 | 不需要爽点节奏和平台预设 |

## 1. 背景和原理 (Background & Principles)

### 1.1 痛点与破局
当前市面上的 AI 写作或互动工具（如 Character.AI 或各类 Copilot）大都深陷于"黑盒打字机"或单调的"角色陪聊"范式中，带来了三大终极痛点：
1. **角色崩坏 (OOC)**：随着剧情深入，AI 容易产生幻觉，遗忘初始设定与核心动机。
2. **节奏拖沓 (缺乏张力)**：AI 习惯顺延上文生成，缺乏主动把握戏剧冲突的能力，极易产生"流水账"。
3. **质感平庸 (缺乏视角与文风)**：现有生成工具无法区分"上帝视角"和"限制视角"，心理描写过于直白，毫无文学留白的美感。

**核心愿景：**
抛弃"AI 代笔 (Copilot)"，拥抱"自主叙事引擎 (Autonomous Narrative Engine)"。用户不再辛苦敲字，而是降临为"上帝 / 导演"，由后台 AI 角色们自动演戏，从而涌现出极具真实感的高能剧情。

### 1.2 运作原理：管线解耦与四大核心 Agent
Genesis Engine 将文学创作解耦为**"发生（剧情逻辑）"**与**"表达（文学渲染）"**两个完全独立的阶段。为保证系统在 MVP (V1.0-V2.0) 阶段的低延迟和 Token 经济性，且贯彻“如非必要、勿增实体”的架构剃刀原则，系统精简为以下**四大核心 Agent**（外加一个确定性的后台提取异步任务）协作驱动：

| Agent | 角色 | 核心职责 | 输入 | 输出 |
|---|---|---|---|---|
| 🧚‍♀️ **The Muse** | 责编/助理 | 面向用户的唯一总代理 (UI Proxy)。翻译口语为底层操作，通过 Markdown 对话确认，拒绝弹窗。 | 用户自然语言 | 结构化操作指令 |
| 🎭 **Character** | 演员本尊 | 每个角色一个独立 AI，基于隐藏动机在沙盒中博弈。 | 场景设定 + Grimoire + RAG 上下文 | 角色意图与动作 |
| 🎛️ **The Maestro**| 指挥家/主控 | **核心逻辑编排器 (Programmatic Orchestrator)。** 放弃不可控的纯自主 Agent，改为由后台严密控制的 for 循环。Maestro (作为 LLM) 在每一回合后仅安全地输出“合法性/张力”打分与“是否收束”的结构化决策。 | 推演回合日志 + Grimoire | 统一的 JSON 结构回复 |
| 🎥 **Camera** | 渲染大师 | 唯一负责文学修饰。根据 POV 和文风参数，将无聊的 Story IR 渲染为最终散文。 | Story IR + 渲染参数 | 文学正文 |
| 📜 **Scribe** | 史官记忆 | 严格基于 Story IR（绝不看最终渲染正文）提取事实写入 DB，保证低级模型防幻觉。 | Story IR Block | Grimoire 向量/属性更新 |

## 2. 界面布局与导航 (UI Layout & Navigation)

### 2.1 整体布局：三栏 + 导航条

![Genesis Engine 3-Pane UI Layout](/docs/genesis_engine_final_ui_1773144668812.png)

系统采用**极简的三栏布局**，UI 交互的最高原则为 **"永不弹窗遮挡、专注唯一稿纸"**。

| 区域 | 名称 | 占比 | 核心功能 |
|---|---|---|---|
| 最左侧极细导航条 | Navigation Strip | ~3% | 切换四大全局视图 |
| 左栏 | The Compass | ~20% | 大纲树 (Storyboard) + 设定树 (Grimoire) |
| 中栏 | The Manuscript | ~52% | 正文编辑器 (上) + 推演监视器 (下) |
| 右栏 | The Muse | ~25% | AI 助理聊天面板（从上到下完整占满） |

### 2.2 四大全局视图 (Navigation Strip)
最左侧的导航条包含四个图标，点击后**切换中栏和左栏的内容**，右栏 The Muse 始终常驻。

| 图标 | 名称 | 功能描述 | 用户可执行的操作 |
|---|---|---|---|
| 🧭 | **Compass (罗盘)** | **默认主工作台**。左栏显示大纲树和设定树，中栏显示正文编辑器和推演监视器。 | 浏览/展开大纲和设定，编辑正文，启动推演 |
| 🕸️ | **Network (关系网)** | **全局实体关系可视化**。中栏替换为力导向图 (Force Graph)，展示角色、势力、地点的动态关系。 | 查看关系、点击节点高亮关联实体、按章节时间线筛选关系快照 |
| 📚 | **Archive (档案馆)** | **全局检索与历史**。中栏替换为搜索界面，支持按角色名、关键剧情事件、章节范围进行全文检索。 | 搜索历史 IR 记录、查看废弃的平行宇宙分支、浏览系统操作日志 |
| ⚙️ | **Settings (设置)** | **项目配置**。中栏替换为设置面板。 | 管理 API 密钥、配置模型路由策略（High/Low）、查看 Token 消费统计看板 |

### 2.3 左栏交互规则

**The Storyboard (大纲树)：**
*   点击某个 `Chapter` 节点 → 中栏 (The Manuscript) **瞬间无缝切换**为该章节文本。
*   点击某个 `Branch` 节点 → 中栏原地切换为该平行宇宙分支的文本。
*   所有切换均为原地替换，用户始终面对"唯一一张稿纸"。

**The Grimoire (设定树)：**
*   单击某个实体节点（角色/地点/势力）→ 节点在左栏**原地向下展开 (Accordion Inline Peek)**，滑出微缩属性卡片（头像、核心属性、当前状态）。中栏纹丝不动。
*   所有设定的修改必须通过 The Muse 的对话确认完成，设定树本身为**只读展示**。

## 3. 创作工作流与用户体验 (The Genesis User Journey)

### 3.1 场景一：零基础开局 (The Cold Start)
对于有着顶级脑洞但厌恶写大纲的作者来说，面对空白项目是最痛苦的。在 Genesis Engine 中，由于 **The Muse Agent** 的存在，这个过程被降维成了一场轻松的"贴身责编催稿局"。

1.  **灵感碰撞 (The Brainstorm)**：用户在 **The Muse** 对话框中直接输入口水话："我想写个赛博朋克修仙。主角是个搞笑道士，修仙靠赛博义体和物理超度。" **The Muse** 会高速解析并引导用户确认核心冲突与主角动机。
2.  **世界观暗中固化 (The Grimoire Injection)**：**The Muse** 在对话流中用 Markdown 格式展示属性修改摘要并询问确认。用户回复"确认"后，属性即刻入库至 **The Grimoire**。
3.  **大纲枝干生成 (The Storyboard Generation)**：**The Muse** 协助构思第一章冲突。确认后，左侧 The Storyboard 树上瞬间挂载新章节，并生成一个 **The Spark**（火花），等待启动推演。

### 3.2 场景二：日常连载流 (The Serialized Workflow)
由于采用了**边演边建 (Lazy Instantiation)** 策略，不需要先建完世界观，就可以开始常态化的连载循环。

#### 阶段一：The Grimoire (世界观状态继承)
作者登入系统，左侧栏的 Grimoire 已自动更新为最新的世界状态。所有世界状态均以不可变的 **Snapshot (快照链)** 自然叠加，属性改变均由后台 **Scribe 异步提炼任务** 在上一次推演提交后自动从 Story IR 中提炼生成，保证角色绝不 OOC。

#### 阶段二：The Spark (创意火花)
**定义：** The Spark 是一段**自然语言冲突指令 + 声明的 Beat 类型**，它是启动一次推演的钥匙。

| 属性 | 说明 |
|---|---|
| **输入位置** | 可在中栏 Storyboard 新节点中直接输入，也可通过 The Muse 对话代为生成 |
| **内容** | 一段口语描述（例："乌家截断了蚕丝，苏檀儿急疯了向宁毅求助，我要宁毅用极度散漫的态度随口给出一条毒计"） |
| **beat_type** | **必选枚举**：`SHOW_OFF_FACE_SLAP`(装逼打脸) / `PAYOFF`(爽点兑现) / `SUSPENSE_SETUP`(悬念铺垫) / `EMOTIONAL_CLIMAX`(情感升华) / `POWER_REVEAL`(金手指展示) / `REVERSAL`(反转) / `WORLDBUILDING`(世界观补完) / `DAILY_SLICE`(日常流) |
| **target_char_count** | 可选（默认 3000）。Camera 渲染时作为硬约束，±10% 内通过 |
| **系统行为** | Muse 解析指令、识别 beat_type、自动从 Grimoire 中拉取相关角色和场景约束，组装后下发至推演沙盒 |
| **触发** | 用户点击 **[Action ▶]** 按钮，推演启动 |

> **为什么换掉"张力评分"？** 网文读者不看"张力"，看"爽点有没有兑现、打脸有没有到位、钩子有没有留下"。把抽象的张力评分换成声明式的 beat_type，Maestro 的判据从"戏够了吗"变成"**装逼打脸的瞬间有没有明确文字**"，LLM 答得准得多，作者也能对齐语言。

#### 阶段三：The Monitor (推演态：异步回合制)
后台的 Agent 们开始基于当前世界状态的沙盒推演。推演结果以**异步回合制**展示在中栏底部的 Monitor 面板中。

**回合 (Turn) 定义：**
*   一个 Turn = 一个 Character 发出动作 + **The Maestro** 评估合法性与张力打分。
*   每个 Turn 完成后，系统暂停 2-3 秒（可在 Settings 中配置），供用户检视。

**场景结束条件 (Beat Completion) 与 防死锁 (Deadlock)：**
*   **The Maestro 按 beat_type 判定**：每回合后，Maestro 基于 Spark 声明的 `beat_type` 专项检查：
    *   `SHOW_OFF_FACE_SLAP` → 检查是否有"主角秀操作"+"反派被挫"的明确文字
    *   `PAYOFF` → 检查铺垫事件是否已在本 Beat 明确兑现
    *   `SUSPENSE_SETUP` → 检查是否已埋下可追溯的未解冲突
    *   `EMOTIONAL_CLIMAX` → 检查是否出现角色情感外化的关键对白/动作
    *   `POWER_REVEAL` → 检查金手指是否已展示并被见证
    *   `REVERSAL` → 检查是否出现明确的形势翻转
    *   `WORLDBUILDING` → 检查是否有新的世界观信息曝光（≥2 条 Grimoire 条目新增）
    *   `DAILY_SLICE` → 达到目标字数即可收束，无硬性戏剧要素要求
*   **环境强制打断 (防死锁)**：如果超过 5 个 Turn 未达成 beat_type 的核心要素，Maestro 将强行抛出突发事件（如："系统事件：门外突然传来刺客脚步声"）。
*   **用户手动终止**：用户随时可以点击 **[Cut ✋]** 按钮，强行终止当前推演段落。

**🕊️ 上帝之手 (God's Hand)：**
用户随时可以点击 Monitor 中某条推演记录：
1.  推演时间线**瞬间冻结**，所有 Agent 暂停。
2.  用户可以**覆写**该角色的意图（如编辑文本："让宁毅态度再狂一点"）。
3.  用户点击 **[Release ▶]**，时间线恢复。后续所有 Agent 的反应将基于覆写后的新设定自动重新推算。

**异常流 (Exception Flows)：**
*   **网络断联/请求超时 (ERR_NET_TIMEOUT):** Monitor 中该角色卡片变灰闪烁，The Muse 弹出提示：“演员 [X] 暂时失去连接，是否跳过其回合或重试？” 恢复路径：点击 **[Retry (重试)]** 或 **[Skip (跳过)]**。
*   **Maestro 裁定动作非法 (ERR_WORLD_REJECT):** Monitor 中该条动作被标记为 ❌，附带 Maestro 驳回理由（如“角色当前没有该武器”）。恢复路径：系统默认自动要求该 Character 重新思考；若连续失败2次，用户可点击 **[Override (上帝特赦)]** 强行通过。

#### 阶段四：IR 脱水与锁定 (The Maestro)
当推演条件满足时（`is_beat_complete: true`），**The Maestro** 最后一次调用会直接输出严谨的结构化剧本中间件 (**Story IR Block JSON**)。此时剧情骨架全部锁死，不可篡改。此环节与上一阶段合二为一，对用户完全透明。

#### 阶段五：全局渲染与局部精修 (The Render & Revision)

**全局渲染台 (Render Mixer)**：位于中栏 The Manuscript 的**右上角常驻入口**。

| 参数 | 类型 | 说明 |
|---|---|---|
| **视点 (POV)** | 下拉选择 | 可选值：某特定角色的限制视角、全知视角（上帝视角）、第一人称 |
| **文风 (Style)** | 下拉 + 自定义 | 内置**平台预设模版**（起点/番茄/晋江/纵横/七猫）+ 通用类型模版（权谋肃杀/仙侠飘逸/都市写实 等）；支持粘贴一段参考文段作为自定义文风锚点 |
| **潜台词 (Subtext)** | 滑块 (0-100%) | 控制留白与心理描写的密度。0% = 纯白描，100% = 极致意识流 |
| **目标字数 (target_char_count)** | 数字 | **网文特化**。默认取 Chapter 节点的设置（通常 3000），Camera 将字数作为渲染硬约束 |
| **最大句长 (max_sent_len)** | 数字 | **平台特化**。番茄/七猫等短句平台必需；默认 30，切平台自动变 |

**渲染约束与守卫（V1.1 新增）：**

1. **字数硬约束**：Camera 输出后自动计数。
    *   少于目标 10% 以上 → 自动追加环境/心理描写（不加情节，不破坏 IR）
    *   多于目标 20% 以上 → 自动压缩（保留对白，砍描述）
    *   ±10% 以内 → 通过
2. **钩子守卫 (Ending Hook Guard)**：Camera 渲染完成后，由低价模型判定最后 200 字：
    *   是否有悬念 / 未决冲突 / 反转 / 角色抉择？
    *   **没有的话**，Camera 只重渲染结尾段（不改前文）。
    *   守卫可在 Settings 关闭（默认开启）。
3. **水字数检测 (Padding Detector)** ：Camera 输出后跑本地启发式：
    *   同一事件被描述超过 3 次 → 标红提示
    *   单段纯环境描写超过 500 字 → 警告
    *   对白占比低于 20% → 警告（网文黄金对白比 30-50%）
    *   **只提示不自动改**，作者自行决定。

*   **全局渲染:** 设定以上参数后，点击 **[Render All]**，Camera Agent 将整章 IR Block 一次性渲染为风格统一的正文。
*   **🔍 局部批改 (Block Revision)**：选中某段不满意的文本 → 弹出 **[Revision (批改)]** 悬浮菜单 → 输入修改指令（如"增加潜台词描写"）→ 仅该段文字在原位无损重写，不影响前后文风。
*   **✏️ 事实修订通道**：选段右键 → **[告诉 Muse 这里改了事实]** → 弹出对话 "你改了什么？" → Muse 生成 Grimoire 的 patch diff → 确认后以**软层 delta** 打补丁到当前快照（不改历史）。**这是作者想改一句话剧情但不想弃掉整段重抽的唯一正确通道。**

**异常流 (Exception Flows)：**
*   **生成被风控拦截 (ERR_SAFETY_BLOCK):** UI 立即停止流式输出，正文区域显示红色下划线，The Muse 悬浮提示：“触碰禁忌法则，渲染被高维虚空拦截（风控过滤）”。恢复路径：点击 **[退回修改 Spark]** 或 **[更换温和语气重试]**。
*   **渲染质量不佳/AI味重:** 用户对整段渲染不满意时，可点击 **[Retry]** 重新调用 Camera Agent。恢复路径：底层 IR 不变，支持无副作用无限重试；若连续不满意，The Muse 将主动建议调整 Render Mixer 参数或切换更高级的基座模型。

#### 阶段六：确认与记忆回写 (The Commit)
1.  用户阅读成文满意后，点击 **[Commit (锁定)]**。
2.  **Scribe Agent** 严格读取阶段四的 Story IR（注意：绝不读取阶段五渲染后的文学散文，以避免幻觉污染），提取最新事实，回写至 The Grimoire。
    *   *架构降级说明 (V1.0)：系统暂不保证用户在编辑器中对“最终渲染正文”进行的手动增删改能反向同步至全局记忆。如需修正剧情事实，请通过撤销本段落重抽 Spark 或微操推演层进行。*
3.  创作闭环完成。用户继续抛出下一个 Spark，推进今日的第二个章节段落。

### 3.3 The Muse 可执行操作清单 (Command Matrix)
**The Muse** 不只是聊天机器人，它是系统暴露给用户的 **终极交互代理 (UI Proxy)**。

**双模设计（V1.1 新增）**：为避免小模型在工具密度过高时幻觉，Muse 内部分两档：
*   **✍️ 写稿档（默认）**：Spark 生成、推演微操、渲染参数、查询、卡文救急。暴露 5-6 个工具。
*   **📚 设定档**（打斜杠 `/设定` 或 UI 小切换）：实体 CRUD、分支管理、回档、事实修订。暴露另外 5-6 个工具。

路由可显式切换或由用户输入前几个词隐式识别。

### 3.4 全本协作的终极奥义 (The Long Game)
1. **世界观自动生长**: The Grimoire 随着每一次推演自动更新，它是真正"边演边建"的，作者无需手动维护人物关系表格。
2. **多分支树状宇宙 (Branching Tree UI)**: 作者可以随时在几百章之后**切出一个新分支（平行宇宙）**，尝试新的剧情走向，且一键回档，所有底层关系网一并重置。不同分支互不干扰。
3. **出书级一键渲染**: 连载结束后，一键修改全局渲染参数（如切为"第三人称全知/剧本对白风"），即可得到不同媒介（剧本、实体书）的全本内容。无需重构一丝一毫的故事逻辑。

> **⚠️ 架构限制声明:** 由于底层采用 **快照链 (Snapshot Chain)** 数据结构，系统**不支持分支合并**功能。每个分支是独立的平行宇宙，无法将两条分支的故事线合并到同一条时间线上。这是架构层面的固有限制，而非功能缺失。如需"合并"剧情，请手动将一条分支的内容复制到另一分支。

## 4. 版本路线图 (Release Roadmap)

### 4.1 V1.0：The Core Engine (核心引擎)
*   **目标:** 跑通"推演 → 脱水 → 渲染"的核心数据链路，验证管线解耦的可行性。
*   **交付物:**
    *   Story IR 的结构化提取与校验系统。
    *   Block 排序引擎（支持追加与任意插入）。
    *   Scribe 后台异步提炼任务（Story IR → The Grimoire 快照链的自动提取写入）。
    *   Camera Agent 基础渲染管线。
*   **用户感知:** 用户在一个相对原始的界面中输入设定 (Spark)，后台通过日志展示推演过程，编辑器中直接呈现渲染后的文字。没有华丽的动画，**但逻辑已经能支撑写完一本连贯的小说。**
*   **验收标准:** 能在系统内完成一个"三角色、五章节"的连贯短篇，Scribe 提取的世界状态在每章结束后 100% 准确。

### 4.1.5 V1.1：The Web Novel Workshop (网文作坊版) 🎯 当前交付目标

*   **目标:** 在 V1.0 核心引擎基础上，用**最少的新功能**让中阶网文作者能真正日更。不追求花哨 UI，追求**日更不断更**。
*   **交付物:**
    1.  **beat_type 枚举系统**：Spark 声明类型，Maestro 按专项判据判完成（替代通用张力评分）。
    2.  **章节字数硬约束**：`target_char_count` 字段，Camera 渲染后自动补/压缩至 ±10%。
    3.  **钩子守卫 (Ending Hook Guard)**：章末 200 字悬念检测，不合格只重渲染结尾。
    4.  **Muse 写稿档 / 设定档分离**：降低工具幻觉率，日常日更只用 5-6 个工具。
    5.  **[卡文救急] 按钮**：一键生成 3 个 Spark 候选，3 个不同方向。
    6.  **事实修订通道**：选段右键 → Muse 打 Grimoire 软层 patch。
    7.  **Scratchpad 落盘**：推演日志追加到 `scratchpad.jsonl`，崩了能恢复。
    8.  **平台预设（起点/番茄/晋江）**：Settings 一选，Render Mixer 默认值全换。
    9.  **VoiceSignature 规则检查**：口头禅 / 称谓 / forbidden_words，Scribe 跑完章节做 grep 级校验。
    10. **日更仪式感 UI**：顶部字数进度条 + 连胜计数器 + Muse 开场问候。
*   **用户感知:** 打开项目 Muse 问候"昨天写到第 47 章，今天想怎么开场？" → 选 Spark → 推演 → 渲染（字数精准命中 3000） → 钩子自动守卫 → Commit → 连胜 +1 🔥。**全流程 60-90 分钟**。
*   **验收标准:**
    *   **🎯 核心验收**：产品负责人本人能用这个版本连续日更 7 天，每天 3000+ 字，无断更，无崩溃丢稿。

### 4.2 V2.0：The Co-Pilot (造梦控制台)
*   **目标:** 在 V1.1 日更能跑的基础上，把产品打磨到**入门作者零门槛**。

## 5. 非功能性需求 (Non-Functional Requirements)

### 5.1 Token 经济学（网文硬指标）

*   **多模型切分降本 (High/Low Routing):** 推演/记忆/规则判定使用低成本模型（Qwen-turbo / DeepSeek-chat）；Character / Camera 渲染层可切换中高端模型（Qwen-plus / DeepSeek-reasoner / GPT-4.6）。
*   **单章成本硬指标:** 默认模型组合下，单章推演+渲染成本必须 ≤ **¥1**（约 30-60k token）。超标时 Settings 红色警告。
*   **前瞻式成本看板（V2.0）:** Settings 预测"按当前节奏本章约 ¥X"，不是事后统计。

### 5.2 Prompt 工程

*   **零硬编码 (No Hardcoded Prompts):** 所有 Agent 的性格设定与工作指令均为独立模板文件集中维护（`backend/prompts/*.jinja2`），可随时热更换无需重部署。
*   **Prompt 中文优先:** Character / Camera 的 system prompt 必须使用中文撰写（防止 LLM 把中文输出带上英文逻辑的腔调）。

### 5.3 数据与架构

*   **不可变数据与单体架构 (Snapshot Chain / SQLite Monolith):** 
    *   快照链（Snapshot Chain）支持 V3.0 任意点回档与多分支，摒弃全局 UPDATE 覆盖。
    *   **软层 Delta (V1.1 新增)**：作者手动事实修订以 delta 形式挂在当前快照，Commit 下章时合并。防止"手改事实 vs Scribe 自动提取"的 race。
    *   全项目统一为 `单一 .sqlite 文件`，支持离线漫游迁移。
    *   上下文强制滑动窗口（Core Memory + Archival Summary + Working Window），V2.0 补 FTS5 + sqlite-vec 混合检索。

### 5.4 稳定性（网文作者死线）

*   **Scratchpad 落盘 (V1.1 必需)**：推演期间每个 Turn Log 追加到 `scratchpad.jsonl`。进程崩溃后启动时扫描未 Commit 的推演，提示用户恢复。
*   **推演可恢复**：Commit 前的中间态不能因为断电/重启丢失。
*   **字数命中率**：Camera 输出字数偏离 target_char_count > 10% 时必须自动补/压缩，不得让作者手动数字数。

### 5.5 性能

*   Monitor 推演日志亚秒级延迟到达前端。
*   Camera 渲染 3000 字正文 ≤ 15 秒。
*   Grimoire 查询 < 100ms；FTS5 全文检索 < 200ms。

### 5.6 安全与 Human-in-the-Loop

*   Human-in-the-loop 是底线。任何对 Grimoire 的**自动写入**必须经 Muse 对话显式确认（Scribe 的 IR→Grimoire 提取除外，这是推演闭环的必然步骤）。
*   **事实修订的软层 delta** 在确认前 100% 可撤销。

### 5.7 平台适配（V1.1 新增 NFR）

*   **平台预设必须内置:** 起点 / 番茄 / 晋江 / 纵横 / 七猫 五平台的 Style + Subtext + max_sent_len 默认值（具体见 `USER_PERSONA.md §1.3`）。
*   **一键平台切换:** Settings 换平台时，所有已有章节的 Render Mixer 默认值同步更新，不影响已渲染内容。

### 5.8 日更仪式感（V1.1 新增 NFR）

*   **开场问候:** 打开项目时 Muse 主动问候，带"昨天写到第 X 章"的上下文和 `[接着写]` 快捷按钮。
*   **连胜计数:** Commit 成功后 UI 显示"连续日更 X 天 🔥"。断更（24h 无 Commit）自动重置。
*   **字数进度条:** 顶部常驻"今日字数 / 目标字数"进度条。
*   **章节里程碑:** 100 / 500 / 1000 / 10000 章成就提醒（非强制，可关闭）。

### 5.9 合规与扩展

*   **i18n / a11y:** **V1.0-V3.0 均不作为阻塞性交付指标**。中文网文市场优先。国际化留给 V3.0 之后或社区。
*   **无账号体系:** 单机单用户，不引入账号、云同步、多租户。作品共享靠 `.sqlite` 文件传递。
